# Расхождения между кодом и ТЗ

## 1. Статус-код успешного checkout

**ТЗ (пример в разделе 7):** `201 Created` для `POST /orders/{id}/checkout`.

**Факт:** в таблице статусов раздела 7 для checkout явно указан `201 Created`, однако контроллер возвращает `200 OK` (`return Ok(new CheckoutResponse(...))`). Для `POST /orders` верно используется `StatusCode(201, ...)`.

**Тест затронут:** `Checkout_Successful_ReturnsCorrectSchema` — ожидает `200`, а не `201`, чтобы соответствовать реальному поведению кода. В ТЗ это расхождение.

---

## 2. C# AddItem: невалидное количество → 400 вместо 422

**ТЗ:** невалидное тело запроса → `422 Unprocessable Entity`.

**Факт:** при `quantity <= 0` в `AddItem` бросается `OrderException`, которое в контроллере маппится на `Problem(400, ...)`. При этом валидация модели ASP.NET (аттрибут `[Range(1, 100)]`) действительно вернёт `422`, но только если значение выходит за пределы 1–100. Если тело вообще некорректное — 422. Если тело валидно по схеме, но нарушает бизнес-правило — 400 вместо ожидаемого 422.

## 3. Лимит количества в AddItemRequest vs PricingService

**ТЗ (раздел 5.5):** максимальное количество одной книги — 20.

**Факт:** в `AddItemRequest` стоит аттрибут `[Range(1, 100)]` — допускает до 100 на уровне HTTP-валидации. `PricingService.ValidateOrder` отловит превышение 20 только при checkout, а не при добавлении товара. Это значит, что через API можно добавить книгу в количестве 50, получить `200 OK`, и только при попытке оформить заказ получить `422`.
